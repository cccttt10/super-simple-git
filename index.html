create a new repo

cd into a directory
create a new git repo
git init

clone a repo

clone a remote repo from GitHub
git clone <url>

workflow

your local repo has 3 "trees"
maintained by git behind the scene
1. Working Directory contains the actual files
2. Index (aka Staging Area)
3. Commit history, where HEAD points to the last commit

add & commit changes

add specified changes to Index
git add <filename>
or add all changes
git add *


commit the added changes
git commit -m "commit message"
now the changes are committed to HEAD
but not pushed to your remote repo yet

push changes

push committed changes to your remote repo
git push origin master
change master to whatever branch you want to push to

if your local repo is not connected to a remote repo
you need to add a remote
git remote add origin <url>
note origin is just a name
you can call it anything
now you can push to your remote repo

branching

use branches to develop features isolated from each other
master is the default branch when you create a repo
use other branches when you develop features
then merge them back to master upon completion

create a new branch named "feature"
and checkout (i.e. swtich to) the branch
git checkout -b feature
or create a new branch without checking out
git branch feature

show all branches and highlight the active branch
git branch -a

switch back to master
git checkout master
delete the feature branch
git branch -d feature

a local branch is not available to other people
unless you push the branch to your remote repo
git push origin <branch>

you can also remember the remote branch
git push -u origin <branch>
so that you can use the following shorthand
for all subsequent pushes
git push

update & merge

sync your local branch with your remote branch
if your local branch "remembers" your remote branch, then
git pull
if not, then
git pull origin master
the pull command will fetch and merge remote changes

to merge another branch into the active branch, use
git merge <branch>


in both pull and merge commands
git tries to auto-merge changes
unfortunately, sometimes auto-merge fails
which results in conflicts
you must resolve them manually
by editing the files that contain conflicting changes
after you finish editing
you need to add your edits
git add <filename>


before merging, you can also compare 2 branches
git diff <source_branch> <target_branch>

tagging

it is standard practice to tag software releases
create a new tag named 1.0.0
git tag 1.0.0 1b2e1d55aa
where 1b2e1d55aa stands for
the first 10 characters of the commit id

log

inspect repo history
git log
you can add parameters to log
e.g. see only the commits authored by Chuntong:
git log --author=chuntong
see a compressed log where each commit is one line:
git log --pretty=oneline
see a decorated tree graph:
git log --graph --oneline --decorate --all
see only which files have changed:
git log --name-status

discard changes

warning
dangerous commands
exercise caution



in case you messed up
which for sure never happens
you can discard changes
git checkout -- <filename>
whcih resets the specified file to HEAD state
changes already added to Index
and untracked (i.e. new) files
will not be affected



if you want to drop ALL your local changes AND commits
fetch the latest history from the remote repo
git fetch origin
and point your master branch to your remote branch
git reset --hard origin/master
