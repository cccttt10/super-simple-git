create a new repo

create a new directory
cd into the directory
create a new git repo by executing
git init

checkout a repo

create a copy for a local repo by executing
git clone <path>
when cloning from GitHub, execute
git clone <url>

workflow

your local repo has 3 "trees"
which are maintained by git behind the scene
1. Working Directory contains the actual files
2. Index (aka Staging Area)
3. HEAD points to the last commit

add & commit changes

propose changes (i.e. add to Index) using
git add <filename>
git add *
this is the first step in the basic git workflow
to commit these changes, execute
git commit -m "commit message"
now the changes are committed to HEAD
but not pushed to your remote repo yet

push changes

your changes are now in HEAD of your local repo
to upload these changes to your remote repo, execute
git push origin master
Change master to whatever branch you want to push your changes to

if your local repo was not cloned from a remote repo
and you want to connect your local repo to a remote repo
you need to add a remote
git remote add origin <url>
note "origin" is just a name and you can call it anything
now you can push changes to your remote repo

branching

use branches to develop features isolated from each other
master is git's default branch when you create a repo
use other branches when you develop features
then merge them back to master upon completion

create a new branch named "feature" and switch to it
git checkout -b feature
switch back to master
git checkout master
delete the feature branch
git branch -d feature_x
a local branch is not available to other people
unless you push the branch to your remote repo
git push origin <branch>

update & merge

to sync your local repo with your remote repo, execute
git pull
the pull command will fetch and merge remote changes
to merge another branch into the active branch, use
git merge <branch>


in both cases git tries to auto-merge changes
unfortunately, this is not always possible
and could result in conflicts
when conflicts arise, you must resolve them manually
by editing the files shown by git
after resolving, you need to mark your edits as merged with
git add <filename>


before merging changes, you can also preview them
git diff <source_branch> <target_branch>

tagging

it's standard practice to tag software releases
create a new tag named 1.0.0
git tag 1.0.0 1b2e1d63ff
where 1b2e1d63ff stands for
the first 10 characters of the commit id
that you want to tag

log

study repo history using
git log
you can add lots of parameters to the log command
e.g. see only the commits authored by chuntong:
git log --author=chuntong
see a compressed log where each commit is one line:
git log --pretty=oneline
see a tree-like, decorated graph of all the branches:
git log --graph --oneline --decorate --all
see only which files have changed:
git log --name-status
for more, see
git log --help

discard changes

warning
dangerous commands
exercise caution



in case you messed up
which for sure never happens
you can discard local changes with
git checkout -- <filename>
this resets your working tree to HEAD
changes already added to Index
as well as untracked (i.e. new) files
will be kept



if you want to drop ALL your local changes AND commits
fetch the latest history from the remote repo
git fetch origin
and point your master branch to your remote branch
git reset --hard origin/master
